11.07.59
 Shastra 🐛 Perbaikan inspectIsolate job error/activity logs to the job (v2.175.1) #53 ▾
The Job Inspector "Errors" and "Activity" tabs leaked global logs from other jobs and domains, and rendered "Invalid Date" for system entries. Root cause: useJobLogs called GET /api/logs?traceId=<jobId>, but Karsa's logs query schema expects snake_case `trace_id`. Zod stripped the unknown `traceId` key, so no trace filter was applied and the endpoint returned the latest 200 logs across ALL jobs/domains. That same endpoint also returns snake_case fields (created_at), so `new Date(log.createdAt)` was NaN -> "Invalid Date". Fix: - Switch useJobLogs to the trace-scoped endpoint GET /api/logs/trace/:jobId (path param parsed server-side), so logs are isolated to this job. This endpoint also returns camelCase fields (createdAt/traceId) via formatLog, fixing the timestamps. - Guard timestamp rendering in JobInspector (mergeActivityItems + formatLogTime helper) so any missing/bad time shows "—" instead of "Invalid Date".
